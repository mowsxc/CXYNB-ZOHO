改动前必须完整验证
- Date: 2026-06-27
- Context: 修改月份添加界面时只改了部分HTML，遗留了预览/确认按钮等旧元素，导致用户看到的还是旧版。后续删除粒子代码时 oldString 范围过大，误删了中间的 PIN 验证函数和 eager render 代码，导致双击 LOGO 功能失效
- Instructions:
  - 修改功能时，用 grep 搜索所有相关元素（HTML、CSS、JS），确保全部更新
  - 提交前检查：删除的元素是否还有引用、新增的元素是否完整、CSS类名是否一致
  - 不要只改一行就提交，要检查整个功能链路是否闭环
  - 用户说"简化为X"时，确认所有旧元素（按钮、状态、预览区）都已移除
  - **删改操作前，评估 oldString 匹配范围风险**：检查要删除的代码块上下是否有其他功能代码，确保 oldString 精确匹配目标范围，不夹带中间代码
  - **每次编辑后立即 grep 验证**：改动完成后用 grep 确认目标函数/元素存在、误删的函数已恢复、引用完整性无破坏

编辑后必须进行语法与符号完整性检查
- Date: 2026-06-30
- Context: 多次编辑后出现括号不匹配、标签未闭合、变量未声明先使用等基础错误，反复触发线上 bug。用户要求每次修改后主动检查而非等报错
- Category: 工作流协作
- Instructions:
  - **每次修改所有文件后**，运行完整校验：
    - Python 文件：`py_compile.compile(doraise=True)`
    - SW/JS 文件：用 Python 的 `re` 统计 `{` `}` 和 `(` `)` 数量平衡（不要用 `node --check`，它不理解 SW 全局变量）
    - SVG 文件：检查 XML 标签闭合
    - HTML 文件：统计 `<div>`/`</div>`、`<script>`/`</script>`、`<style>`/`</style>` 标签对平衡
  - 检查变量声明先于使用（特别是 `var` hoisting 在条件分支中的陷阱）
  - 字符串拼接生成的 HTML，逐个 `+=` 追踪开闭标签对应关系
  - **在回复"修复完成"之前运行以上检查**，不依赖用户发现

优先评估而非盲目修复
- Date: 2026-06-30
- Context: 审计出 29 个问题时，对 `_data_cache` 无大小限制等"问题"做了实际影响分析而非全盘修复，用户表扬这种思考方式
- Instructions:
  - 列出修复清单时，对每项评估实际风险等级和数据量影响
  - 数据量小（如 6 个月 × 30 天）的内存占用不是问题，强行加限制反而增加复杂度
  - 外部不可控的边缘场景（如"两个月份 URL 相同"）标注为低风险而非高优先级
  - 区分"会影响行为的 bug"和"理论风险"
